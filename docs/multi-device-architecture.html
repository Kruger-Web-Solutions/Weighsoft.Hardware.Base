Multi-Device Architecture

How the Serial Reader, Display, and Serial Writer ESP32 firmware variants compose into larger systems. Each ESP runs a server side and a client side; the diagrams below trace the wires between them.

Sheet 00 · Device Reference

Device Reference

Each ESP32 firmware variant runs a server side and a client side. The cards below summarize what each device serves and what it connects out to.

Each device is a separate firmware build maintained on its own git branch — branches are not merged. The cards below identify the source location and branch for each variant.

R-01

Serial Reader

Reads from a physical serial device (scale, sensor) and exposes the live stream to any subscriber.

Server
  • GET/POST /rest/serial
  • WS /ws/serial
  • MQTT publish weighsoft/serial/{id}/data
  • BLE read+notify 12340000-…-1234
Client
  • WeightForwarderService HTTP POSTs lines to a remote URL (e.g. Node-RED).

src/examples/serial/SerialService.{h,cpp}
src/examples/weightforwarder/WeightForwarderService.{h,cpp}

D-01

Display

Drives a 16x2 I2C LCD. Accepts data from any source; can also pull a remote stream onto the screen.

Server
  • GET/POST /rest/display
  • WS /ws/display
  • MQTT subscribe weighsoft/display/{id}/set
Client
  • DisplayService Serial Bridge connects out to a remote reader's /ws/serial, MQTT topic, or BLE peripheral and mirrors data on the LCD.

branch · display
src/examples/display/DisplayService.{h,cpp}

W-01

Serial Writer

Writes to a physical serial port. Accepts pending lines from any source, or pulls them from a remote reader.

Server
  • GET/POST /rest/serialWriter
  • WS /ws/serialWriter
  • MQTT subscribe weighsoft/serialwriter/{id}/send
  • MQTT publish status weighsoft/serialwriter/{id}/status
Client
  • SerialWriterForwarderService HTTP-polls or WebSocket-subscribes to a remote reader and writes received lines to the UART (and optionally USB CDC).

Sheet A · 01

Multiple Readers → Node-RED

Several Readers, each tapping its own serial device, all forward their stream to a central Node-RED instance.

How it works: Each Reader runs its own WeightForwarderService with the Node-RED HTTP endpoint as its target. Readers don't talk to each other. Node-RED receives one POST per parsed line per reader.

Real-world example: three weighbridges across a yard, all forwarding to a central Node-RED dashboard.

Sheet B · 02

1 Reader + Display + Writer

One Reader at the center. A Display and a Writer both pull data from its server side.

How it works: The Display's Serial Bridge connects out to the Reader's /ws/serial. The Writer's SerialWriterForwarderService HTTP-polls the Reader (the firmware default; WebSocket subscribe is also supported via the same service config).

Real-world example: one scale read by an ESP, mirrored to a floor LCD and echoed onto a printer's serial port.

Sheet C · 03

Reader as local monitor + cloud forwarder

A single Reader's server serves a Display while its own client side (Forwarder) pushes the same stream to Node-RED. Both flows run simultaneously.

How it works: The Reader's SerialService serves the Display over /ws/serial. In the same firmware, WeightForwarderService independently HTTP-POSTs every parsed line to Node-RED. Server side and client side run in parallel inside the same ESP.

Real-world example: local LCD on the production line for the operator, cloud logging in parallel for the office.

Sheet D · 04

Multiple Displays from one Reader

One Reader, many Display ESPs all subscribed to its server. Every screen shows the same value live.

How it works: Each Display's Serial Bridge connects out to the same Reader's /ws/serial. WebSocket fan-out is handled inside the Reader by the framework's WebSocketTxRx — every connected client receives the same broadcast.

Real-world example: one scale read by an ESP, the same value shown on every status board on the warehouse floor.

Sheet E · 05

Node-RED commanding a Writer

Reverse direction. Node-RED POSTs commands to a Writer; the Writer pushes them out a physical serial port.

How it works: Node-RED POSTs {"pending_line": "..."} to the Writer's /rest/serialWriter endpoint. SerialWriterService picks up the queued line on its next loop and writes it out the UART (and optionally USB CDC).

Real-world example: Node-RED triggers a label printer or PLC by POSTing the command string to the Writer.

Sheet F · 06

MQTT broker as hub

All ESPs talk to one MQTT broker instead of each other. The broker decouples publishers and subscribers.

How it works: Reader publishes to weighsoft/serial/{id}/data. Display subscribes to weighsoft/display/{id}/set; in this topology the broker (or a Node-RED bridge node) republishes Reader data onto the Display topic. The Writer subscribes to weighsoft/serialwriter/{id}/send and publishes status on /status. Node-RED can subscribe to any of these.

Real-world example: all ESPs talk to one broker; replacing or adding a device requires no IP reconfiguration on the others.

Sheet G · 07

Serial WiFi bridge

Two physical serial ports linked across WiFi. Reader reads device A; Writer writes the same data to device B.

How it works: The Writer's SerialWriterForwarderService is configured with the Reader as its source URL. The Writer pulls every line and re-emits it on its own UART. The dashed segment is the WiFi hop; dots traverse it continuously so the visual reads as one stream from device A to device B.

Real-world example: two pieces of legacy serial equipment that need to share data but are physically too far apart for an RS-232 cable.

Sheet H · 08

Mixed protocol fan-out

One Reader, four consumers, four protocols — all running concurrently from the same source line.

How it works: Each consumer uses the protocol best suited to it. Display subscribes via /ws/serial. Writer's Forwarder HTTP-polls the Reader. Node-RED gets HTTP POSTs from the Reader's WeightForwarderService. A mobile app reads the Reader's BLE notify characteristic. The pink Mobile/BLE consumer appears only here.

Real-world example: one reader, four consumers — a single source of truth feeding every part of the operation in the protocol that suits each consumer.
