Multi-Device Architecture

How the Serial Reader, Display, and Serial Writer ESP32 firmware variants compose into larger systems. Each ESP runs a server side and a client side; the diagrams below trace the wires between them.

Sheet 00 · Device Reference

Device Reference

Each ESP32 firmware variant runs a server side and a client side. The cards below summarize what each device serves and what it connects out to.

Each device is a separate firmware build maintained on its own git branch — branches are not merged. The cards below identify the source location and branch for each variant.

R-01

Serial Reader

Reads from a physical serial device (scale, sensor) and exposes the live stream to any subscriber.

Server
  • GET/POST /rest/serial
  • WS /ws/serial
  • MQTT publish weighsoft/serial/{id}/data
  • BLE read+notify 12340000-…-1234
Client
  • WeightForwarderService HTTP POSTs lines to a remote URL (e.g. Node-RED).

branch · serialReader
src/examples/serial/SerialService.{h,cpp}
src/examples/weightforwarder/WeightForwarderService.{h,cpp}

D-01

Display

Drives a 16x2 I2C LCD. Accepts data from any source; can also pull a remote stream onto the screen.

Server
  • GET/POST /rest/display
  • WS /ws/display
  • MQTT subscribe weighsoft/display/{id}/set
Client
  • DisplayService Serial Bridge connects out to a remote reader's /ws/serial, MQTT topic, or BLE peripheral and mirrors data on the LCD.

branch · display
src/examples/display/DisplayService.{h,cpp}

W-01

Serial Writer

Writes to a physical serial port. Accepts pending lines from any source, or pulls them from a remote reader.

Server
  • GET/POST /rest/serialWriter
  • WS /ws/serialWriter
  • MQTT subscribe weighsoft/serialwriter/{id}/send
  • MQTT publish status weighsoft/serialwriter/{id}/status
Client
  • SerialWriterForwarderService HTTP-polls or WebSocket-subscribes to a remote reader and writes received lines to the UART (and optionally USB CDC).

branch · serialWriter
src/examples/serialwriter/SerialWriterService.{h,cpp}
src/examples/serialwriter/SerialWriterForwarderService.{h,cpp}

Sheet A · 01

Multiple Readers → Node-RED

Several Readers, each tapping its own serial device, all forward their stream to a central Node-RED instance.

How it works: Each Reader runs its own WeightForwarderService with the Node-RED HTTP endpoint as its target. Readers don't talk to each other. Node-RED receives one POST per parsed line per reader.

Real-world example: three weighbridges across a yard, all forwarding to a central Node-RED dashboard.
