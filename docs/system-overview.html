Serial Reader Platform

ESP32/ESP8266 firmware for reading serial instrument data, extracting values via regex, broadcasting to 4 channels, and forwarding to remote targets.

Architecture

System Overview

Four services share a single Serial1 hardware port. UartModeService arbitrates who owns it. SerialService reads and broadcasts. WeightForwarderService relays to remote systems. DiagnosticsService tests the hardware.

Layer 1

Hardware Interface

Serial1 (hardware UART) connects the ESP32 to any instrument that outputs data over RS-232 / TTL serial — weight scales, PLCs, sensors, barcode readers.

GPIO17 (RX2)Serial1 receive — data from instrument
GPIO18 (TX2)Serial1 transmit — future command output
3.3 V logicUse level shifter for 5 V TTL sources
LoopbackBridge GPIO17↔GPIO18 for diagnostic tests

Layer 2

Firmware Services

Four services each follow the StatefulService<T> pattern — thread-safe state, automatic channel propagation, and LittleFS persistence.

📡

SerialService

Reads bytes from Serial1 into a 2048-char line buffer. On newline: enqueues line, applies POSIX regex to extract weight, publishes to all 4 channels at ≤4 lines/sec.

src/examples/serial/SerialService.cpp
🔬

DiagnosticsService

Hardware testing suite: loopback echo test, automatic baud rate scanner (1200–115200), and signal quality analyser (latency + jitter over 1000 packets).

src/examples/diagnostics/DiagnosticsService.cpp
🔀

UartModeService

State machine with two modes: LIVE_MONITORING hands Serial1 to SerialService; DIAGNOSTICS suspends SerialService and frees the port for test packets.

src/UartMode.h
📤

WeightForwarderService

Listens to SerialService update callbacks. Rate-limited to 4 forwards/sec. Supports four outbound protocols: HTTP POST, WebSocket client, MQTT publish, BLE write.

src/examples/weightforwarder/WeightForwarderService.cpp

Build Targets

PlatformIO Environments

Four hardware targets share the same firmware source. BLE is available on all ESP32 targets and disabled on ESP8266.

🎯

node32s (Primary)

LILYGO TTGO / Node32S. Custom OTA partitions. Production target. GPIO17/18 for Serial1.

partitions_ble_ota.csv
🔌

esp32dev

Generic ESP32 DevKit. Same firmware, default partitions. Ideal for initial bring-up and bench testing.

board = esp32dev
⚡

esp32s3

ESP32-S3 — newer silicon with USB-OTG. All features including BLE 5.0. Same GPIO17/18 pinout.

platform = espressif32
📡

esp12e (ESP8266)

ESP-12E at 160 MHz. No BLE. No mutex (single-core). FT_BLE disabled automatically. Reduced WS client capacity.

platform = espressif8266

Frontend

React UI

React 18 + TypeScript + Material-UI 5. Built by ProgmemGenerator into C++ PROGMEM arrays and served directly from ESP32 flash. JWT-authenticated with light/dark theme.

Provider & Routing Tree

Serial Monitor Tabs

Documentation Tab

Static page showing hardware wiring diagram, GPIO pin table, getting-started steps, and POSIX regex syntax guide with examples.

Wiring Table

GPIO17 ← instrument RX · GPIO18 → instrument TX (future). Level-shifter warning for 5 V sources. Link to diagnostics for loopback verification.

Polling Mode

Calls GET /rest/serial every 2 seconds via useRest hook. Displays last_line, extracted weight, timestamp, and debug snapshot fields.

Debug Info

Shows dbg_uart_rx_avail, dbg_suspended, dbg_serial_started, dbg_line_buf_len — useful to verify UART is running and not suspended by DiagnosticsService.

Real-Time Stream

Uses useWs hook (Sockette). Fires on every new line arriving from the instrument. No polling interval — event-driven updates.

Ring Buffer

Keeps last 100 lines in state array. Auto-scrolls to bottom. Monospace font. Timestamps in grey, raw line in white, extracted weight in green.

TypeScript API Layer

interface SerialData {
  last_line:      string;    // Full raw line from instrument
  weight:         string;    // Extracted capture group
  timestamp:      number;    // millis() on ESP32
  baud_rate:      number;    // e.g. 9600
  data_bits:      number;    // 5|6|7|8
  stop_bits:      number;    // 1|2
  parity:         number;    // 0=None 1=Even 2=Odd
  regex_pattern:  string;    // POSIX regex
  dbg_uart_rx_avail?: number; // bytes in UART buffer
  dbg_suspended?:     number; // 1 if DiagnosticsService active
}
