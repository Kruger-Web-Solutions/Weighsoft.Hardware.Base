Overview

The serialWriter branch turns an ESP32 into a serial TX controller. It exposes a REST/WebSocket/MQTT/BLE API to send arbitrary strings out through a physical UART (Serial1), optionally appending a configurable line terminator. It also acts as a Serial Reader Source client — connecting to a remote serialReader device, pulling its live weight string, and forwarding it to Serial1 TX or native USB CDC. Full UART diagnostics (loopback, baud scan, signal quality) share the same Serial1 hardware via UartModeService arbitration.

What This Branch Does

Sends strings out through Serial1 TX (GPIO17) on demand — from the React UI, REST API, WebSocket, MQTT, or BLE. The line terminator (LF / CRLF / CR / NONE) is configurable at runtime. An integrated Serial Reader Source can autonomously pull data from a remote serialReader device and forward it to Serial1 or native USB CDC.

Two Output Sinks

Serial1 TX GPIO17 UART to external instruments. Requires UART mode = Writer. Baud rate, data bits, parity, stop bits all configurable.

Native USB CDC ESP32-S3 USB-OTG port. Carries clean data only — no developer logs. PC-side scale tools connect here directly.

Serial Reader Source

Connects outward to a serialReader device via HTTP or WebSocket, extracts the last_line field, and routes it to either or both output sinks. No manual intervention required once configured.

UART Diagnostics

Three test modes share Serial1 via UartModeService arbitration: Loopback — TX→RX echo quality, Baud Scanner — auto-detect 8 baud rates, Signal Quality — latency / jitter / packet loss over N packets.

Most scales use 3.3V TTL. If your instrument uses RS-232 (±12V), a MAX232 level-shifter is required.

Native USB CDC

The ESP32-S3's USB-OTG peripheral is initialised as a second serial port (USBCDC) reserved exclusively for clean data output. Developer logs continue on the on-board UART0 bridge port at 115200 baud — the two streams never mix.

Requires explicit wiring in main.cpp: serialWriterService->setDataUsbCdc(&dataUsbCdc)

Loopback Jumper (Diagnostics)

To run loopback / baud scan / signal quality tests, bridge GPIO17 TX → GPIO18 RX with a short jumper wire. UartModeService switches Serial1 ownership from SerialWriterService to DiagnosticsService when diagnostics mode is selected.

Onboard LED

LedExampleService controls the built-in LED on GPIO2 via REST/WS/MQTT/BLE. ESP32: active HIGH. Publishes Home Assistant MQTT discovery at boot.

Key Data Flows

Operator types a string in the React UI (or sends via API). The string is set as pending_line; SerialWriterService::loop() picks it up next tick and writes it to Serial1.

// POST to /rest/serialWriter
{ "pending_line": "TARE" }

// WebSocket — send via /ws/serialWriter
{ "pending_line": "ZERO" }

// State update from sent echo (origin="serial_writer_sent" — filtered, no restart):
{ "last_sent_line": "TARE", "sent_count": 42, "pending_line": "" }
Key fix applied: Sending pending_line does not restart Serial1. The onConfigUpdated() guard compares applied vs current baud/config/terminator and skips the 100ms Serial1.end()+begin() when only runtime fields changed.

ForwarderService connects as a WebSocket client to the serialReader's /ws/serial endpoint. It authenticates with JWT if credentials are set, receives JSON frames, extracts last_line, and routes to output sinks.

// Config (POST /rest/serialWriterForwarder)
{
  "enabled": true,
  "protocol": 1,                             // 1 = WebSocket
  "source_url": "ws://192.168.1.xx/ws/serial",
  "json_field": "last_line",
  "output_targets": "serial1_only",          // or "usb_only" | "both"
  "auth_username": "admin",
  "auth_password": "admin"
}

ForwarderService polls the serialReader's REST endpoint on a configurable interval. Each response is parsed for the json_field value.

// Config for HTTP mode
{ "protocol": 0, "source_url": "http://192.168.1.xx/rest/serial",
  "json_field": "last_line", "poll_interval_ms": 500 }

When UART mode is switched to Diagnostics, UartModeService calls serialWriterService->suspendSerial() and hands Serial1 ownership to DiagnosticsService. Three test modes are available.

Switch mode via POST /rest/uartMode {"mode":"diagnostics"} or the UartMode switcher in the React UI.

2. Stored in browser localStorage

3. All AXIOS requests inject Authorization: Bearer … via transformRequest

4. WebSocket appends ?access_token=… query param

5. 401 response → Axios interceptor → redirect /unauthorized → sign out

Serial Reader Source Auth

When pointing the Serial Reader Source at a serialReader device that requires a login (IS_AUTHENTICATED on /ws/serial), provide auth_username + auth_password in the config. The forwarder service signs in to the remote device and manages token refresh automatically.

The password is never returned by GET /rest/serialWriterForwarder. Send auth_password_clear: true in a POST to explicitly remove the stored password.

Key Hooks & Patterns

useRest<T>

import { useRest } from '../../utils';
const { data, saveData, saving, setData } =
  useRest({ read: readSerialWriter,
            update: updateSerialWriterConfig });

Auto-loads on mount via GET. saveData() calls POST, shows snackbar.

useWs<T>

import { useWs } from '../../utils/useWs';
const { connected, data } =
  useWs<SerialWriterData>(WEB_SOCKET_ROOT + 'serialWriter');

Sockette with JWT ?access_token= query param. 100ms debounced sends. Optimistic updates when origin_id matches own clientId.

FeaturesContext

const { features } = useFeatures();
// features loaded from GET /rest/features at boot
// Never use import.meta.env for feature flags!

Flags are runtime — reflect what's actually compiled in the firmware. BLE tab only visible when features.ble === true.

AXIOS (always)

import AXIOS from '../../api/endpoints';
// baseURL = /rest/ · JWT injected automatically
AXIOS.post('serialWriter', { pending_line: 'TARE' });

Always import AXIOS — never fetch() or plain axios. The instance injects Authorization: Bearer … via transformRequest.

Gotchas

Fixed: pending_line POSTs no longer restart the UART.
onConfigUpdated() compares the applied baud/serial-config/terminator against the new values before calling applySerialConfig(). A pending_line POST changes no hardware parameters → early return, no 100ms Serial1.end()+begin() blocking the loop.

Only actual config changes (baud rate, data bits, parity, stop bits, line terminator) trigger a Serial1 restart and flash write.

USB CDC sink must be wired in main.cpp.
Without serialWriterService->setDataUsbCdc(&dataUsbCdc), the USB sink is unconfigured and lines are silently dropped (one-shot warning to Serial). The wiring is already present in src/main.cpp but must not be removed.

The ESP32-S3 has two USB hardware blocks: the on-board UART0 bridge (logs at 115200) and the native USB-OTG peripheral (clean data). They operate independently.

Always use AXIOS, never fetch().
The AXIOS instance from '../../api/endpoints' injects Authorization: Bearer … automatically. Plain fetch() sends no auth header and gets a 401.
Firmware upload = web interface update.
The React build is embedded in the firmware binary via PROGMEM_WWW. After upload: press the physical EN/RST button, then Ctrl+Shift+R in the browser. 
pio run --target upload
# → press EN/RST on board
# → Ctrl+Shift+R in browser
BLE partition has no OTA slots.
partitions_ble.csv is a single-app layout. You cannot use ArduinoOTA or HTTP firmware upload with this partition. Always use USB serial upload for BLE builds.
