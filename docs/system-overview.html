Overview

The display branch turns an ESP32 (or ESP8266) into a smart I²C LCD controller. It drives a 16×2 character LCD via LiquidCrystal_I2C, exposes a REST/WebSocket/MQTT/BLE API for controlling what the LCD shows, and — uniquely — can act as a Serial Bridge client: it connects to a remote Weighsoft serialReader device and automatically mirrors whatever that device's serial line reads onto the physical LCD screen.

What This Branch Does

Drives a 16×2 character LCD via I²C. Accepts text content from any channel (REST, WebSocket, MQTT, BLE). Optionally bridges a remote serialReader device — connecting back to its WebSocket/MQTT/BLE feed and rendering live weight data on the physical LCD.

Serial Bridge Modes

WebSocket Connects to /ws/serial on a remote device IP. Receives {"last_line":"…"} JSON frames.

MQTT Subscribes to a configured topic. Expects same JSON frame format.

BLE Scans for the remote device's BLE service UUID, connects, registers NOTIFY on its characteristic.

Key Characteristics

  • No filesystem persistence — all state is RAM only (except framework configs)
  • BLE operates as both server (for apps) and client (to bridge remote device)
  • I²C address configurable at runtime (0x27 or 0x3F)
  • LCD preview rendered in the React UI in real time

I²C Address

Most YwRobot / PCF8574-based backpacks ship at 0x27. Some modules use 0x3F.

The address is configurable at runtime via the Control tab or REST API — the firmware will re-initialise the LCD driver at the new address without reboot.

VCC is 5V. The I²C lines are pulled up to 5V on the PCB. ESP32 GPIO is 3.3V-tolerant on input but may need a level shifter for long runs. Short bench wires work fine in practice.

Built-in LED

LedExampleService controls the onboard LED on GPIO2.

ESP32: active HIGH. ESP8266: active LOW (inverted logic).

Exposed via /rest/ledExample and /ws/ledExample. Home Assistant MQTT discovery is auto-published at boot.

Data Flows

User writes text directly to the display via REST or WebSocket — no external device involved.

REST flow

POST /rest/display
Authorization: Bearer <jwt>
Content-Type: application/json

{ "line1": "12.500 kg", "line2": "NET     ", "backlight": true }

WebSocket flow

// Client sends:
{ "line1": "12.500 kg", "line2": "NET     " }

// Server broadcasts to all WS clients:
{ "type": "payload", "origin_id": "<clientId>", "payload": { "line1": "…", "line2": "…", … } }

DisplayService connects as a WebSocket client to the serialReader device's /ws/serial endpoint. Incoming JSON frames update the LCD.

Bridge is configured via the Serial Bridge tab in the UI. The display device needs a route to the serialReader's IP on port 80.

// Serial Bridge config (POST /rest/display)
{
  "bridge_mode": "websocket",
  "serial_device_ip": "192.168.1.xx",
  "serial_device_port": 80
}

Both display and serialReader devices share a common MQTT broker. SerialReader publishes weight data; Display subscribes and renders it.

// Bridge config
{
  "bridge_mode": "mqtt",
  "serial_mqtt_topic": "weighsoft/serial/<serial-device-id>/data"
}

The display ESP32 acts as a BLE client: it scans for the serialReader's BLE service UUID, connects, and subscribes to NOTIFY on the weight characteristic.

// Bridge config
{
  "bridge_mode": "ble",
  "serial_ble_service_uuid": "12340000-e8f2-537e-4f6c-d104768a1234",
  "serial_ble_char_uuid":    "12340001-e8f2-537e-4f6c-d104768a1234"
}

JWT Flow

1. POST /rest/signIn with username + password → returns { access_token: "…" }

2. Token stored in localStorage

3. All AXIOS requests inject Authorization: Bearer <token> via transformRequest

4. WebSocket connections append ?access_token=<token> query param

5. 401 response → Axios interceptor redirects to /unauthorized → sign-out

Key Hooks

useRest<T>

Import from '../../utils'

const { data, setData, saveData, saving } =
  useRest<DisplayData>({
    read:   readDisplayData,   // GET /rest/display
    update: updateDisplayData  // POST /rest/display
  });

Auto-loads on mount. saveData() calls POST, shows snackbar on success/error.

useWs<T>

Import from '../../utils'

const { connected, data, updateData } =
  useWs<DisplayData>(DISPLAY_SOCKET_PATH);
// DISPLAY_SOCKET_PATH = ws://<host>/ws/display

Sockette with JWT ?access_token= query param. 100ms debounce on sends. Optimistic updates when origin_id matches own clientId.

FeaturesContext

Import from '../../contexts/features'

const { features } = useFeatures();
if (features.ble) { /* show BLE tab */ }

Features loaded from GET /rest/features at app start. Never use import.meta.env for feature flags — they're runtime, not build-time.

AXIOS instance

Import from '../../api/endpoints'

import AXIOS from '../../api/endpoints';
// baseURL = /rest/
// JWT header injected automatically

Always use AXIOS, never fetch() or plain axios — the instance injects the JWT Bearer token via transformRequest.

Gotchas

Always use AXIOS, never fetch()
The AXIOS instance imported from '../../api/endpoints' injects Authorization: Bearer <token> via transformRequest before every request. Plain fetch() sends no auth header → 401 Unauthorized. 
// ✓ Correct
import AXIOS from '../../api/endpoints';
AXIOS.post('display', data);

// ✗ Wrong — gets 401
fetch('/rest/display', { method: 'POST', body: JSON.stringify(data) });
Feature flags are runtime, not build-time
Flags come from GET /rest/features at app start. They reflect what was compiled into the firmware. Do not use import.meta.env — those are Vite build-time values and won't match the actual firmware capabilities. 
// ✓ Correct
const { features } = useFeatures();
{features.ble && <BleTab />}

// ✗ Wrong
{import.meta.env.VITE_BLE_ENABLED && <BleTab />}
Hook import path is '../../utils'
useWs and useRest live in interface/src/utils/. Trying to import from '../../hooks' or '../../components' will fail at build time. 
// ✓ Correct
import { useWs, useRest } from '../../utils';

// ✗ Wrong
import { useWs } from '../../hooks/useWs';
import { useRest } from '../../components/useRest';
Firmware upload = web interface update
The React build is embedded inside the firmware binary via PROGMEM_WWW. Uploading to the filesystem does NOT update the web UI. After every firmware upload you must: (1) upload the firmware, (2) press the physical EN/RST reset button, (3) hard-refresh the browser (Ctrl+Shift+R). 
pio run --target upload   # uploads firmware.bin containing the React app
# Then: press EN/RST on the board
# Then: Ctrl+Shift+R in browser
COM port busy on Windows
If PlatformIO upload fails with "port is busy", a Python serial monitor process is holding the COM port. Kill it before uploading. 
# PowerShell — kill serial monitor processes
Get-Process python* | Stop-Process -Force

# Then retry upload
pio run --target upload
BLE partition has no OTA slots
The default node32s environment uses partitions_ble.csv — a single-app layout that gives BLE more heap by eliminating the OTA partitions. You cannot use ArduinoOTA or the HTTP firmware upload with this partition. Always use USB serial upload for BLE builds.
